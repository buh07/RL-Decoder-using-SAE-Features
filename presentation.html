RL-Decoder with SAE Features

Complete Research Story: What We Tried, What Worked, and What Did Not

Main question: when a model writes step-by-step reasoning (CoT), is that text actually describing how the model got the answer?

We built a full test pipeline. It starts with basic checks, then measures internal arithmetic features, then tests whether those features can detect faithful vs unfaithful reasoning text.

Plain definitions: CoT = Chain-of-Thought text. SAE = sparse autoencoder used to turn hidden states into interpretable sparse features. AUROC = ranking quality score (0.5 random, 1.0 perfect).
Snapshot: as of March 7, 2026.
Project state: GPT-2 Phase 7 is closed. Track C is a negative result under this protocol. Qwen is now a separate next-model test.
Final GPT-2 Phase 7 config: text=0.50, latent=0.50, confidence=0.0, causal=0.0, with structural penalties enabled.

Phases 4-5 Are arithmetic features real, and can we control the model with them? The features are real and causally active for arithmetic outputs, but simple steering with those features is not stable.

Phase 1: Ground-Truth Validation (Tooling Sanity Check)

Intuition

Before touching LLM reasoning, first prove SAE can recover hidden variables when we already know the correct answer.

Method
  • Used controlled environments: BFS traversal, stack-machine, and logic puzzle traces.
  • Each environment has known internal state labels.
  • Trained SAEs and checked whether decoded features recover those labels.
Key Results
  • R^2 range across tasks: 0.967-0.999.
  • All tasks passed strong reconstruction and decoding checks.
  • No major instability across seeds.
Impact On Next Phase

Because this passed, we had evidence the method itself works. That justified moving to full GPT-2 layer-wide SAE training.

Phase 2 / 2r: Multi-Layer SAE Training and TopK Stabilization

Intuition

To understand reasoning internals, we need features from every layer, not only one or two layers.

Method
  • Phase 2: trained SAEs across all GPT-2 layers.
  • Phase 2r: retrained with TopK activation so sparsity is controlled directly.
  • Saved per-layer stats and checkpoints for downstream probes and interventions.
Key Results
  • 24 GPT-2 SAEs at 12x expansion.
  • 98 SAEs across 4 models in broader assets.
  • TopK setup stabilized around ~30% active feature rate for GPT-2.
  • Training quality was stable enough to standardize later experiments.
Impact On Next Phase

This gave us a stable feature space for tracing where model information appears across tokens and layers.

Phase 3: Reasoning Flow Tracing

Intuition

Before designing probes, we mapped where feature activity happens during reasoning text.

Method
  • Captured hidden states from every GPT-2 block during reasoning generation.
  • Encoded token states with per-layer SAEs.
  • Built token-layer activity maps for equation, intermediate, and non-computation tokens.
Key Results
  • Average occupancy roughly ~50% active / ~50% sparse.
  • Simple active-count differences between useful and non-useful tokens were small.
  • This showed we needed targeted features, not coarse counts.
Impact On Next Phase

We moved from coarse counting to targeted arithmetic probes tied to equation positions and step roles.

Phase 4 / 4r: Arithmetic Probing and Causal Subspace Tests

Intuition

If arithmetic is represented internally, we should be able to decode arithmetic state and causally perturb that state.

Method
  • Used GSM8K traces with arithmetic annotations at equation and result tokens.
  • Trained ridge probes to predict arithmetic targets from SAE features.
  • Ran causal subspace patching (Phase 4r) to test whether those features affect output probability.
Key Results
  • Probe performance: around R^2=0.977 at layer 7 (TopK stack).
  • Causal subspace result: peak +0.107 delta_logprob at layer 22.
  • 21/24 layers showed positive directional effect in 4r causal profile.
  • Plain meaning: arithmetic features are present and can matter causally.
Impact On Next Phase

This justified building a faithfulness auditor, but first we had to test whether these features are controllable in practice.

Phase 5: Feature Interpretation and Steering Stress Test

Intuition

A feature can predict behavior but still fail as a stable control handle. We tested that directly.

Method
  • Generated top-feature context cards for interpretability.
  • Applied mean-difference steering vectors across layers and measured output shifts.
  • Compared expected direction vs observed direction.
Key Results
  • Interpretation showed broad high-activation arithmetic-associated features.
  • Mean-diff steering produced uniformly negative shifts (disruptive, not useful control).
  • Control behavior was unstable even when probes looked strong.
Impact On Next Phase

We stopped assuming "good probe = good control." In later phases we used these channels as evidence signals, not guaranteed interventions.

Phase 6: Decoder Benchmark + Full Layer Sweep

Intuition

Before Phase 7 claims, we needed strong decoder baselines and robust checkpoint coverage.

Method
  • Ran full supervised sweep over input modes (raw/hybrid/sae) and layer windows.
  • Used fixed split policy and leakage guards.
  • Selected parity checkpoints for the final faithfulness matrix.
Key Results
  • Sweep completion: 102/102 configs, 0 failed.
  • Best top-1: raw_block8_00_07 = 0.5628.
  • Best top-5: hybrid_block4_04_07 = 0.7860.
  • This gave a stable base for fair Phase 7 comparisons.
Impact On Next Phase

We entered Phase 7 with reliable checkpoints, standardized artifacts, and strong leakage controls.

Phase 7 (GPT-2): Main Question, Rules, and Final Outcome

Main question

Can internal model signals tell us whether a reasoning trace is faithful, not just whether the text looks plausible?

  • Track A: text pattern signal.
  • Track B: decoded latent-state signal.
  • Track C: causal/mechanistic signal.

Rules we enforced

  • Leakage-safe trace-group split with fixed seed.
  • Calibration and evaluation kept separate.
  • Variant-robust checks on hard cases.
  • Bootstrap CIs and grouped cross-validation before closure.

Final GPT-2 deployment

  • text=0.50, latent=0.50
  • confidence=0.0, causal=0.0
  • Structural penalties enabled.
  • Track C closed as a negative result for GPT-2.
Bottom line for this phase: we ran many corrected versions of Track C, but none passed robust closure criteria on GPT-2.

Phase 7 Progression: Step-by-Step With Rationale

Step Why we ran it Result Decision after result
1) Core matrix (4 checkpoints x 4 variables) Start with the full baseline test under one fixed protocol. Track C did not show robust pass behavior. Do not conclude yet. First fix known engineering risks.
2) Causal bug-fix reruns Make sure failures are not caused by indexing, lookup, or token-anchor bugs. Code path improved and diagnostics became cleaner, but core Track C weakness stayed. Treat this as scientific signal, then run targeted reformulations.
3) R1 confidence-margin track Test whether decoder confidence gives a better faithfulness signal than hard matches. Some signal, but not strong enough for closure. Keep as diagnostic channel, continue to stronger tests.
4) R2 trajectory-coherence track Test whether step-to-step consistency can separate faithful vs unfaithful traces. Direction was wrong (anti-predictive). Reject this formulation for GPT-2.
5) R3 contrastive probe Test if a learned classifier over hidden-derived features can recover Track C signal. Early run looked promising, but robust follow-up was not stable enough. Proceed to stricter fairness and robustness tests.
6) Option B fair profile comparison Verify gains are real by re-auditing each profile, not just blending scores. P1/P2/P3 were very close; required ordering test failed. No strong evidence that added Track C weighting is helping.
7) Path C robust validation Stress test on hard variant with train exclusions + grouped CV. Single split near threshold, pooled CV below target. Treat as not robust.
8) Mixed hidden + SAE ladder Check if combining feature spaces adds independent signal. No stable incremental gain beyond SAE block. Close GPT-2 Track C negative under current protocol.
This progression matters: each new step was chosen to answer a specific failure mode from the previous step.

Phase 7 SAE Experiment: Design and Rationale

Data used

  • Used existing control-conditioned records from Phase 7.
  • These records already include per-step hidden states at the equation anchor.
  • No new data collection was needed for the SAE experiments.

Core SAE setup

  • For each layer, normalize hidden states with saved activation stats.
  • Encode into sparse SAE feature vectors (12,288 dimensions).
  • Run all tests on those feature vectors, not only raw hidden states.

Why this was important

  • Raw L2 can hide small but useful feature differences.
  • Sparse features let us test if specific dimensions separate faithful and unfaithful traces.
  • This directly tested whether Track C failed because of representation choice.

SAE Experiment Ladder

SAE step What we measured Why we measured it
SAE feature discrimination Feature-space L2, per-feature divergence, sparse logistic probe, overlap with Phase 4 features Check if single-step SAE features carry faithfulness signal.
SAE trajectory coherence Step-to-step cosine smoothness, variance coherence, monotonicity coherence Check if multi-step feature dynamics catch wrong reasoning chains.
Path B feature-set swaps Compare result_top50, eq_pre_result_150, divergent_top50 Test whether wrong feature subset caused weak hard-variant performance.
Path C robust validation Variant-robust wrong_intermediate gate with bootstrap and grouped CV Separate real signal from structural confounds and split noise.
Mixed hidden + SAE Model ladder (SAE-only vs mixed) with incremental gain checks Test if combining spaces adds independent signal.

Why Phase 7 Is Important (Academic Value)

A clean contrast

  • Earlier phases proved arithmetic features are real and causally active.
  • Phase 7 showed those same features do not robustly verify CoT faithfulness in GPT-2.
  • This is a strong and informative scientific contrast.

Better research discipline

  • We did not stop at first failure or first success.
  • We fixed bugs, reran fair comparisons, then applied robust validation.
  • This reduces the chance of false claims from noisy runs.

Clear claim boundaries

  • Operational detection can still be strong.
  • Mechanistic faithfulness verification is a higher bar and was not met here.
  • This makes the final paper claims honest and useful.
Question Answer after Phase 7
Did we prove GPT-2 has arithmetic internals? Yes.
Did we prove Track C robustly verifies GPT-2 CoT faithfulness? No.
Is this still publishable and meaningful? Yes. The positive mechanism result plus negative faithfulness result is a strong contribution.

Research Contribution (Why This Matters)

Result 1: Internal arithmetic is real

  • Strong decodability of arithmetic state from internal features.
  • Positive subspace intervention signal in upper layers.
  • This is direct evidence the model stores structured arithmetic information.

Result 2: Internal arithmetic does not imply faithful CoT

  • Track C remained negative/non-robust for GPT-2 under strict protocol.
  • This shows a clear gap between "model can compute" and "text explains that computation."

Result 3: Practical detection is still useful

  • Text + latent + structural penalties gives strong practical discrimination (Option B profile range around 0.836-0.838 AUROC).
  • Good for operational filtering, but not proof of mechanistic faithfulness.
